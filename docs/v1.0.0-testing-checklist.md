# v1.0.0 Testing Checklist

Pre-release manual verification. Each section covers one of the four reworks plus the broader admin / enrichment surfaces. Work through it on the dev install, mark items as you go.

## Preconditions

- [ ] **FluentCRM 3.0+** installed and active
- [ ] **FluentCRM AI** configured: provider = Claude, valid Anthropic API key, `is_enabled = yes`
- [ ] **Anthropic org-level web search** is enabled (Claude Console → privacy)
- [ ] **At least one company** with contacts attached, for the company-side tests
- [ ] **At least one contact** with `individual_research_consent = Allowed` (the default)
- [ ] **Local copy of the migration notice precondition:** if you have an `fce_api_key` option still set from pre-1.0.0 testing, the migration notice will fire; if not, the notice tests are N/A

---

## 1. Provider integration

### Health states (Dashboard tab)
- [ ] **Ready (Claude)** — Dashboard "Your setup" panel shows green check on AI provider with the model name; green check on Web search
- [ ] **Disabled** — set `is_enabled = no` in FluentCRM → Settings → AI Configuration. Plugin Dashboard shows a yellow warning row on AI provider with a link back to FluentCRM AI settings. The warning banner appears at the top of non-Dashboard tabs.
- [ ] **Not configured** — clear the API key in FluentCRM AI settings. Dashboard shows red error on AI provider. Banner appears on non-Dashboard tabs.
- [ ] **Unsupported provider** — set FluentCRM AI to OpenAI or Gemini. Dashboard shows yellow warning on Web search ("not enabled for this provider"). Enrich runs should fail with a clear "this release supports Claude only" message.

Restore FluentCRM AI to Claude + enabled before continuing.

### Enrichment using the bridge
- [ ] **Trigger a contact enrichment** — click Enrich on a contact profile, watch for "Queued" status; reload after ~30s, status should be "Complete" with web-search citations in the four-section note
- [ ] **Trigger a company enrichment** — same flow on a company profile
- [ ] **Confirm the API call uses FluentCRM's key** — temporarily set an invalid key in FluentCRM AI settings, run an enrichment, confirm it fails with the FluentCRM-configured error rather than a stored-plugin-key error. Restore the real key.

### Migration notice
- [ ] **If `fce_api_key` exists from a pre-1.0.0 install:** notice renders on every admin page. Verify it shows the decrypted key behind a "Show legacy key" disclosure, the "I've migrated — dismiss" button, and the "Delete legacy key permanently" button.
- [ ] **Dismiss the notice** for the current user. Verify it doesn't reappear on reload for that user.
- [ ] **Other admins still see the notice** — log in as a different admin (or just verify the dismiss is in `wp_usermeta`, not an option).
- [ ] **Delete permanently** — click the destructive button. Verify `fce_api_key` is removed (via WP-CLI: `wp option get fce_api_key` returns empty).

## 2. Company module detection

Set up:
- [ ] In FluentCRM → Settings → Experimental, **note the current Company Module state** (probably ON for your dev install).

### Module ON
- [ ] Plugin Dashboard "Your setup" shows green check on Company module
- [ ] Tab navigation shows: Dashboard, Contact Context, **Company Context**, Focus Areas, Capacity Tiers, **Danger Zone**
- [ ] Contact profile sidebar shows the 8 `org_*` mirror fields plus enrichment status fields plus individual fields (a contact's "Custom Profile Data" sidebar in FluentCRM admin)

### Module OFF
- [ ] In FluentCRM → Settings → Experimental, **disable the Company Module**
- [ ] Reload the plugin Dashboard. "Your setup" shows yellow warning on Company module
- [ ] Tab navigation now shows: Dashboard, Contact Context, Focus Areas, Capacity Tiers (no Company Context, no Danger Zone)
- [ ] **Try a deep-linked URL** like `?page=fluentcrm-contact-enrichment#/company-context` — should redirect to Dashboard since the route is hidden
- [ ] Contact profile sidebar **no longer shows the 8 `org_*` fields** (existence in DB intact; just not rendered)
- [ ] Contact enrichment still works (the individual_* fields populate normally)
- [ ] Company profile section is not registered (the FluentCRM company-list page should just not exist when the module is off — there's nothing to test here directly)

### Module flip back ON
- [ ] Re-enable Company Module in FluentCRM Experimental settings
- [ ] Reload the plugin Dashboard. "Your setup" goes back to green
- [ ] **The 8 `org_*` mirror fields reappear** on contact profiles (existing stored values still present — no data loss)
- [ ] Company Context and Danger Zone tabs reappear

### Routing

- [ ] **Edit a context module**, switch to the Lookup Fields subtab → **no prompt** (same parent tab; save state shared).
- [ ] **Select a different module in the list rail** → **no prompt** (same save context; just a query-param change).
- [ ] After **a successful Save**, navigating anywhere doesn't prompt (dirty cleared).

### Visual quality

- [ ] **Layout is responsive** — narrow the browser; the Modules subtab stacks rail above editor below 900px.

## 5. REST API surface

For each endpoint, easiest way: use the Vue UI which exercises them naturally. But for explicit verification:

- [ ] `GET /wp-json/fce/v1/focus-areas` returns `{ options: [...] }`
- [ ] `POST /wp-json/fce/v1/focus-areas` with `{ options: [...] }` saves and returns the cleaned list
- [ ] `GET /wp-json/fce/v1/capacity-tiers` returns `{ options: [...], defaults: [...] }`
- [ ] `POST /wp-json/fce/v1/capacity-tiers` with an empty `options` array falls back to defaults
- [ ] `GET /wp-json/fce/v1/context-modules/contact` returns `{ modules: [...] }`
- [ ] `POST /wp-json/fce/v1/context-modules/contact` with `{ modules: [...] }` saves
- [ ] `GET /wp-json/fce/v1/lookup-fields/contact` returns `{ available: [...], selected: [...] }`
- [ ] `GET /wp-json/fce/v1/meta-prompt/contact` returns `{ prompt: "..." }`
- [ ] `POST /wp-json/fce/v1/bulk-resync` with `{ confirmation: "RESYNC" }` returns counts; without RESYNC returns 400
- [ ] **Permission check**: log in as a non-admin user and confirm any endpoint returns 403

## Known limitations to verify behavior on (not bugs)

- [ ] **OpenAI/Gemini active in FluentCRM** → plugin shows "supports Claude only" message; enrichment runs fail cleanly. (Adapters land in v1.0.1.)
- [ ] **FluentCRM AI disabled** → plugin shows "AI is disabled" message; enrichment runs fail cleanly.
- [ ] **Web search disabled at Anthropic org level** → enrichment runs return without citations and structured fields may be sparser. (No way for us to detect this client-side; the failure mode is silent and the existing prompt handles it gracefully.)
- [ ] **Bundle size** (~700KB gzipped): admin page first load is heavier than before; subsequent loads are cached.
